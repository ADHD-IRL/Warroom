# WARROOM — Replit Build Prompt

## Full-Stack Application Specification

-----

## OVERVIEW

Build a full-stack web application called **WARROOM** — a multidisciplinary AI-powered red-team and strategic analysis platform. The application allows users to:

1. Build and manage a library of **Agents** (expert archetypes with discipline profiles, biases, and focus areas)
1. Design **Chains** (multi-step compound threat or scenario sequences that link agents)
1. Organize everything into **Domains**, **Scenarios**, and **Threats**
1. Run **Sessions** — structured two-round debates where agents analyze a scenario independently (Round 1) and then debate each other (Round 2)
1. Generate agent assessments, chain narratives, and synthesis reports using the **Anthropic Claude API**

The aesthetic should be **dark, editorial, and serious** — like a military operations center crossed with a premium intelligence briefing tool. Think dark navy backgrounds, amber/gold accents, sharp typography, and data-dense layouts that feel operational, not playful.

-----

## TECH STACK

- **Frontend**: React + Vite
- **Styling**: Tailwind CSS with a custom dark theme
- **State**: Zustand for global state management
- **Database**: SQLite via Drizzle ORM (Replit DB or file-based)
- **Backend**: Express.js API server
- **AI**: Anthropic Claude API (`claude-sonnet-4-20250514`) — use the `ANTHROPIC_API_KEY` environment variable
- **Icons**: Lucide React
- **Routing**: React Router v6

## DATABASE SCHEMA

Create the following tables:

### `domains`

```
id, name, description, color (hex), icon, created_at, updated_at
```

*Examples: “Defense Acquisition”, “Energy Security”, “Cybersecurity”, “Geopolitics”*

### `agents`

```
id, name, discipline, domain_id (FK), persona_description, cognitive_bias, 
red_team_focus, severity_default (CRITICAL/HIGH/MEDIUM/LOW), 
vector_human (0-100), vector_technical (0-100), vector_physical (0-100), 
vector_futures (0-100), is_ai_generated (bool), tags (JSON array),
created_at, updated_at
```

### `scenarios`

```
id, name, description, domain_id (FK), context_document (long text),
status (draft/active/archived), tags (JSON array), created_at, updated_at
```

### `threats`

```
id, name, description, domain_id (FK), scenario_id (FK nullable),
severity (CRITICAL/HIGH/MEDIUM/LOW), category, tags (JSON array),
created_at, updated_at
```

### `chains`

```
id, name, description, domain_id (FK), scenario_id (FK nullable),
threat_id (FK nullable), is_ai_generated (bool), tags (JSON array),
created_at, updated_at
```

### `chain_steps`

```
id, chain_id (FK), agent_id (FK nullable), step_number, 
agent_label (text — e.g. "Agent 3 (Supply Chain)"),
step_text, created_at
```

### `sessions`

```
id, name, scenario_id (FK), domain_id (FK), status (pending/round1/round2/complete),
phase_focus (text — e.g. "Design Phase"), created_at, updated_at
```

### `session_agents`

```
id, session_id (FK), agent_id (FK), round1_assessment (long text),
round1_severity, round2_rebuttal (long text), round2_revised_severity,
round2_strongest_ally_agent_id (FK nullable), round2_strongest_disagree_agent_id (FK nullable),
compound_chain_text (long text), generated_at
```

### `session_findings`

```
id, session_id (FK), finding_code, severity, finding_text, 
agent_consensus_count, is_new (bool), created_at
```

### `session_synthesis`

```
id (1:1 with session_id), session_id (FK), 
consensus_findings (JSON), contested_findings (JSON),
compound_chains (JSON), blind_spots (JSON),
priority_mitigations (JSON), sharpest_insights (JSON),
generated_at
```

- Total Agents | with breakdown by domain

- Chains in Library | with count by domain
- Scenarios | with draft/active counts

**Recent Sessions Panel**: Last 5 sessions with status badges, domain, date, quick “Continue” button

**Agent Coverage Matrix Widget**: Mini version of the coverage heatmap — rows = top 8 most-used agents, columns = the 8 lifecycle phases. Uses color intensity.

**Quick Actions**:

- “New Session” (primary CTA — amber button)

### PAGE 2: DOMAINS MANAGER

**Purpose**: Domains are the top-level organizational categories (e.g., “Defense Acquisition”, “Energy Security”, “Cybersecurity”).

#### SESSION WORKSPACE

Full-page workspace that shows the session in progress.

**Top bar**: Session name | Scenario | Phase Focus | Status badge | “Generate Round 1 / Round 2 / Synthesis” buttons

**Layout**: Tabbed interface:

-----

**TAB: ROUND 1 — Independent Assessments**

Agent cards in a 2-3 column grid. Each card:

```
┌─────────────────────────────────────────────┐
│ [Agent Color Bar]                            │
│ Agent N: [Name]              [CRITICAL]      │
│ Discipline                                   │
│ ─────────────────────────────────────────── │
│ [Expand/Collapse]                            │
│                                              │

│                                              │

└─────────────────────────────────────────────┘
```

**Generate All (Round 1)** button at top: Runs all agents in parallel (sequential API calls with progress shown). Each agent card animates as it generates. A progress bar shows X/N complete.

When generating, pass to Claude:

```
You are [agent name], [persona description].
Your cognitive bias: [cognitive_bias]
Your red-team focus: [red_team_focus]

SCENARIO CONTEXT:
{context_document}

PHASE/FOCUS: {phase_focus}

Write a Round 1 independent threat/scenario assessment (600-900 words) covering:
1. Opening position — your primary framing of the situation from your discipline
2. Threat/Finding 1 — with specific mechanism, what defenders/analysts are missing, and severity (CRITICAL/HIGH/MEDIUM) with rationale
3. Threat/Finding 2 — same structure
4. Threat/Finding 3 — same structure
5. Invalidating assumption — one assumption that if wrong would change your entire assessment
6. Key finding — one-sentence bottom line

Write in first person as the expert. Be specific, opinionated, and willing to disagree with conventional wisdom.
```

-----

**TAB: ROUND 2 — Cross-Discipline Debate**

Same card layout, but now each card shows the rebuttal.

The “Generate All (Round 2)” button sends EVERY agent their own assessment PLUS all other agents’ Round 1 assessments.

Prompt:

```
You are [agent name], [persona description].
Your cognitive bias: [cognitive_bias]

You have just read all Round 1 assessments from the other experts. Here they are:

[ROUND 1 ASSESSMENTS FROM ALL OTHER AGENTS — formatted with agent name headers]

Now write your Round 2 rebuttal (400-600 words) covering:
1. Strongest alliance — which other agent's findings combine most powerfully with yours, and build the compound chain that emerges (name the agent explicitly)
2. Strongest disagreement — which agent you disagree with most and exactly why (name them, reference their specific argument)
3. Whether you've revised your severity rating and why
4. ONE compound chain narrative — a multi-step sequence threading your discipline with at least one other agent's discipline

Be direct. Name names. Quote other agents' arguments when disagreeing. Change your position if another agent persuaded you.
```

-----

**TAB: SYNTHESIS**

Auto-generated synthesis after Round 2 is complete.

Shows structured output:

- **Consensus Findings** (sorted by severity)
- **Contested Findings** (agent A vs agent B format)
- **Compound Chains** (extracted from Round 2 rebuttals, with agent attributions)
- **Blind Spots** (areas no agent covered well)
- **Priority Mitigations** (numbered list)
- **Sharpest Insights** (5 best quotes from agents, highlighted in navy boxes)

Each section has an “Edit” button to manually override/supplement.

**Save to Chain Library** button on any compound chain found in synthesis.

**Export** button — generates markdown report.

-----

**TAB: AGENTS MAP**

Visual representation of the session:

- Agent cards arranged in a circle or grid
- Lines drawn between agents who cited each other in Round 2 (strongest ally / strongest disagreement — green vs red lines)
- Chain nodes floating in the space showing which chains connect which agents
- Clicking an agent shows their assessments
- Clicking a chain shows its steps

**TAB: SETTINGS**

- Edit session name, phase focus
- Add/remove agents (if session not yet complete)

- Reset to Round 1

## INTERACTION PATTERNS

### Drag-and-Drop Agent Selection

In Session creation, agent cards in the library are draggable into the session panel. Visual drop zone highlights when dragging. Clicking an agent also adds it via a “+” button.

### Inline Editing

All text fields support click-to-edit inline — no separate “edit mode” required for small fields. Large fields (persona, context documents) open in a modal editor.

### Keyboard Shortcuts

- `Cmd+N` — New item (context-aware: new agent on agents page, new chain on chains page)
- `Cmd+S` — Save current item
- `Escape` — Close modal/flyout

### Optimistic Updates

All create/update operations update UI immediately, then sync to backend. Show error toast and revert on failure.

### Collapsible Sections

Long assessment texts in sessions are collapsed by default (showing first 3 lines + “Show more”). Click to expand.

## IMPORTANT IMPLEMENTATION NOTES

1. **API Key**: Store `ANTHROPIC_API_KEY` as a Replit Secret. Never expose it to the frontend. All Claude API calls go through the Express backend.
1. **Model**: Always use `claude-sonnet-4-20250514` for all generation tasks.
1. **Streaming**: For long agent assessments (Round 1 / Round 2), use streaming responses so users see text appearing as it generates rather than waiting for the full response. Pipe the stream through a Server-Sent Events endpoint.
1. **Rate Limiting**: When generating all agents in a session simultaneously, use a queue with a small delay between API calls (200ms) to avoid overwhelming the API.
1. **Error Handling**: If an agent generation fails, show a “Retry” button on that specific agent card. Don’t fail the entire batch.
1. **Persistence**: Auto-save agent assessments to the database as they complete. If the user navigates away and returns, completed assessments are preserved.
1. **Mobile**: The application is primarily designed for desktop (1200px+). A basic mobile view should collapse the sidebar into a hamburger menu and stack panels vertically, but deep functionality is desktop-only.
1. **Empty States**: Every list/grid page should have a meaningful empty state with an illustration (SVG inline), a description of what goes here, and a primary action button.
1. **Confirmation Dialogs**: Destructive actions (delete agent, delete chain, reset session) require a confirmation modal with the item name typed.
1. **Clone Functionality**: Agents and Chains can be cloned — creates a copy with “ (Copy)” appended to the name and “is_ai_generated” preserved from original.
